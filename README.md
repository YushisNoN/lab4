## Язык программирования
### Синтаксис
Язык представляет собой пародию на **ассемблер** с поддержкой:
- Меток (labels)
- Секций (`.text`, `.data`)
- Директивы `.org` для размещения кода по конкретным адресам
- Констант (`.const`) и строк (`.str`)
- Пользовательские макросы через директиву (`.macro`, `.endm`)
- Условное компилирование через директивы (`.if`, `.ifdef`, `.else`, `.ifend`)

#### Формат инструкций
| Формат | Синтаксис | Пример |
|--------|-----------|--------|
| R-тип | `OPCODE Rd, Rs1, Rs2` | `ADD R1, R2, R3` |
| I-тип | `OPCODE Rd, Rs1, #imm` | `ADDI R1, R2, #10` |
| J-тип | `OPCODE label` | `JMP loop` |
| O-тип | `OPCODE` | `HALT` |

#### Поддерживаемые инструкции

| Мнемоника | OPCODE | Операнды | Схема выполнения |
|-----------|--------|-----------|------------------|
| `NOP` | 0x00 |  - | `PC <- PC + 1` — ничего не делает, только инкрементирует счётчик команд |
| `HALT` | 0x01 |  - | `halt_flag <- 1` — устанавливает флаг остановки, процессор прекращает выполнение |
| `NEG` | 0x02 |  Rd, Rs1 | `ALU <- -Rs1`; `Rd <- ALU`; `flags <- ALU` — инвертирует знак числа в Rs1, результат в Rd |
| `LOAD` | 0x10 |  Rd, Rs1, [Rs2] | `addr <- Rs1 + Rs2`; `Rd <- MEM[addr]` — загрузка из памяти по адресу, вычисленному как сумма двух регистров |
| `LOADI` | 0x11 |  Rd, #imm | `Rd <- imm` — запись непосредственного значения (константы) в регистр |
| `STORE` | 0x12 | Rs1, [Rs2] | `addr <- Rs1 + Rs2`; `MEM[addr] <- Rs1` — сохранение значения регистра в память по адресу Rs1+Rs2 |
| `ADD` | 0x20 |  Rd, Rs1, Rs2 | `ALU <- Rs1 + Rs2`; `Rd <- ALU`; `flags <- ALU` — сложение двух регистров |
| `SUB` | 0x21 |  Rd, Rs1, Rs2 | `ALU <- Rs1 - Rs2`; `Rd <- ALU`; `flags <- ALU` — вычитание регистров |
| `MUL` | 0x22 |  Rd, Rs1, Rs2 | `ALU <- Rs1 * Rs2`; `Rd <- ALU`; `flags <- ALU` — умножение (младшие 32 бита) |
| `DIV` | 0x23 |  Rd, Rs1, Rs2 | `ALU <- Rs1 / Rs2`; `Rd <- ALU`; `flags <- ALU` — целочисленное деление |
| `ADDI` | 0x24 |  Rd, Rs1, #imm | `ALU <- Rs1 + imm`; `Rd <- ALU`; `flags <- ALU` — сложение с константой |
| `SUBI` | 0x25 |  Rd, Rs1, #imm | `ALU <- Rs1 - imm`; `Rd <- ALU`; `flags <- ALU` — вычитание константы |
| `AND` | 0x40 |  Rd, Rs1, Rs2 | `ALU <- Rs1 & Rs2`; `Rd <- ALU`; `flags <- ALU` — побитовое И |
| `OR` | 0x41 |  Rd, Rs1, Rs2 | `ALU <- Rs1 \| Rs2`; `Rd <- ALU`; `flags <- ALU` — побитовое ИЛИ |
| `XOR` | 0x42 |  Rd, Rs1, Rs2 | `ALU <- Rs1 ^ Rs2`; `Rd <- ALU`; `flags <- ALU` — побитовое XOR |
| `MOV` | 0x30 |  Rd, Rs1 | `Rd <- Rs1` — копирование значения одного регистра в другой (без ALU) |
| `CMP` | 0x31 |  Rs1, Rs2 | `ALU <- Rs1 - Rs2`; `flags <- ALU` — сравнение, результат не сохраняется, только флаги |
| `JMP` | 0x50 |  label | `PC <- imm` — безусловный переход на адрес из поля imm |
| `JZ` | 0x51 |  label | `if (Z == 1) PC <- imm` — переход, если флаг Z=1 |
| `JNZ` | 0x52 |  label | `if (Z == 0) PC <- imm` — переход, если флаг Z=0 |
| `JG` | 0x53 |  label | `if (Z == 0 && N == V) PC <- imm` — переход, если больше (знаковое сравнение) |
| `JL` | 0x54 |  label | `if (N != V) PC <- imm` — переход, если меньше (знаковое сравнение) |
| `JGE` | 0x55 |  label | `if (N == V) PC <- imm` — переход, если больше или равно (знаковое сравнение) |
| `PUSH` | 0x70 |  - | `SP <- SP - 1`; `MEM[SP] <- Rs1` — стек растёт вниз, запись регистра в стек |
| `POP` | 0x71 |  - | `Rd <- MEM[SP]`; `SP <- SP + 1` — чтение из стека, стек растёт вверх |
| `CALL` | 0x72 |  label | `SP <- SP - 1`; `MEM[SP] <- PC`; `PC <- imm` — сохранение адреса возврата, переход на подпрограмму |
| `RET` | 0x73 |  - | `PC <- MEM[SP]`; `SP <- SP + 1` — восстановление адреса возврата из стека |
| `PSTR` | 0x74 |  Rs1 | `length <- MEM[Rs1]`; `for i in 0..length-1: OUT <- MEM[Rs1 + i + 1]` — вывод Pascal-строки |
| `IN` | 0x75 |  Rd | `Rd <- MEM[0xFF]` — чтение из порта ввода (MMIO) |
| `OUT` | 0x76 |  Rs1 | `MEM[0xFE] <- Rs1` — запись в порт вывода (MMIO) |

#### Пояснение к выполнению инструкций

Все инструкции выполняются за одну или несколько микроопераций. Процессор имеет следующие основные компоненты:

| Компонент | Обозначение | Назначение |
|-----------|-------------|------------|
| Регистровый файл | `RegFile` | 16 регистров R0-R15, R0 всегда равен 0 |
| Указатель стека | `SP` | Хранит адрес вершины стека (растёт вниз) |
| Счётчик команд | `PC` | Хранит адрес текущей инструкции |
| Регистр инструкций | `IR` | Хранит текущую выполняемую инструкцию |
| АЛУ | `ALU` | Выполняет арифметические и логические операции |
| Флаги | `Z, N, V, C` | Устанавливаются по результату операций в ALU |

**Флаги процессора:**

| Флаг | Название | Условие установки |
|------|----------|-------------------|
| `Z` | Zero | Результат операции равен 0 |
| `N` | Negative | Старший бит результата = 1 |
| `V` | Overflow | Переполнение при знаковой операции |
| `C` | Carry | Перенос при беззнаковой операции |

## Организация памяти

Архитектура — **Гарвардская** (`harv`): память команд и память данных физически разделены.

### Память команд (Instruction Memory)

| Характеристика | Значение |
|----------------|----------|
| Размер | Конфигурируется через JSON (поле `instr_memory_size`) |
| Разрядность | 32 бита |
| Адресация | прямая, абсолютная |
| Содержимое | исполняемый код |

### Память данных (Data Memory)

| Характеристика | Значение |
|----------------|----------|
| Размер | Конфигурируется через JSON (поле `data_memory_size`) |
| Разрядность | 32 бита |
| Тип | однопортовая |

#### Карта памяти данных

| Диапазон адресов | Назначение |
|------------------|------------|
| 0x0000 - 0x00EF | Глобальные переменные и данные |
| 0x00F0 - 0x00FD | Стек (растёт вниз, в сторону младших адресов) |
| 0x00FE | Порт вывода (OUT, MMIO) |
| 0x00FF | Порт ввода (IN, MMIO) |

### Работа со строками (pstr)

Строки хранятся в формате **Pascal-string** (`pstr`):

```text
[длина][char0][char1][char2]...[charN]
```
## Система команд (ISA)

### Кодирование инструкций

Все инструкции имеют фиксированную длину — **1 слово (32 бита)**.

#### Формат инструкций
``` text
 31    24 23   20 19   16 15   12 11     0
+--------+-------+-------+-------+--------+
| OPCODE |  Rd   |  Rs1  |  Rs2  |  imm   |
+--------+-------+-------+-------+--------+
```
Где
`OPCODE` (8 бит): код операции
`Rd` (4 бита): регистр-приемник
`Rs1` (4 бита): регистр источник
`Rs2` (4 бита): регистр источник
`imm` (12 бит): непосредственное значение


## Микрокод (Control Unit)

Управление — **микропрограммное** (`mc`). Каждая машинная инструкция реализована через последовательность микроинструкций, хранящихся в Microcode ROM.

### Структура микроинструкции

Микроинструкция содержит управляющие сигналы для всех компонентов процессора:

| Поле | Тип | Описание | Возможные значения |
|------|-----|----------|-------------------|
| `src_a` | str | Источник для входа A АЛУ | `rs1`, `rd`, `imm`, `SP`, `IN`, `OUT` |
| `src_b` | str | Источник для входа B АЛУ | `rs2`, `imm`, `SP`, `PC` |
| `alu_op` | str | Операция АЛУ | `ADD`, `SUB`, `MUL`, `DIV`, `AND`, `OR`, `XOR`, `NEG` |
| `reg_write` | bool | Запись результата в регистровый файл | `0`/`1` |
| `mem_read` | bool | Чтение из памяти данных | `0`/`1` |
| `mem_write` | bool | Запись в память данных | `0`/`1` |
| `mem_addr` | str | Источник адреса для памяти | `SP`, `IN`, `OUT`, `rs1` |
| `mem_data` | str | Источник данных для записи в память | `rs1`, `PC` |
| `use_imm` | bool | Использовать непосредственное значение | `0`/`1` |
| `write_flags` | bool | Обновить флаги ALU | `0`/`1` |
| `pc_write` | bool | Запись в PC (условный переход) | `0`/`1` |
| `cond` | str | Условие для условного перехода | `Z`, `NZ`, `G`, `GE`, `L`, `LE`, `C` |
| `jump` | bool | Безусловный переход в микропрограмме | `0`/`1` |
| `next_addr` | int/str | Следующий адрес микрокоманды | число или `IMM_JUMP`, `RET_PC`, `PSTR_HANDLER` |
| `sp_dec` | bool | Декремент SP (SP = SP - 1) | `0`/`1` |
| `sp_inc` | bool | Инкремент SP (SP = SP + 1) | `0`/`1` |

### Организация Microcode ROM

Микрокод хранится в памяти объёмом 512 записей. Адресация осуществляется следующим образом:

```text
Адрес микрокоманды = OPCODE * 4 + смещение
```
Где, `OPCODE` (8 бит): код операции, `смещение`: номер микроинструкции (например 0, 1, 2, 3)

### Исполнение инструкции

Процессор моделируется с точностью до такта. Выполнение инструкций разбито на несколько этапов:

#### Fetch (Выборка) (такт 1)

Инструкция, находящаяся по адресу `PC` загружается в регистр `IR`

#### Increment PC (Выборка) (такт 2)
Счётчик команд увеличивается на 1 (получаем адрес следующей инструкции)
`PC <- PC + 1`

#### Decode (Выборка) (такт 3)
Из `IR` извлекается `OPCODE`, вычисляется нчальный адрес микропрограммы 
```text
OPCODE <- IR[31:24]
MP <- OPCODE * 4
```

#### Execute (такты с 4, по N)
Происходит последовательное выполнение микроинструкций, до тех пор, пока не произойдет возврат в главный цикл (то есть счетчик микрокоманд не сбросится до 0) `MP = 0`
```text
micro_step <- MICROCODE[MP]
Исполнение микроинструкции
MP <- next_MP (MP+1 or JMP)
```
#### Finish (последний такт)
Счетчик микрокоманд сбрасывается в ноль `MP=0`

## Ввод-вывод

Ввод-вывод реализован как **поток токенов** (`stream`).

### Модель потока токенов

В данной модели ввод и вывод представлены в виде последовательности токенов (символов), которые обрабатываются процессором последовательно. Сама последовательность задается в конфигуарационном файле в разделе `input`
**Принцип работы:**

1. При старте модели формируется буфер ввода (input buffer) из данных, указанных в конфигурационном файле
2. Процессор читает токены из буфера по запросу (инструкция IN)
3. Токены выводятся в буфер вывода (output buffer) по команде OUT
4. По окончании работы модели буфер вывода сохраняется и отображается

### Буфер ввода (Input Buffer)

**Создание буфера:**
- Буфер создаётся при инициализации модели процессора
- Данные загружаются из поля `input` конфигурационного файла
- Поддерживаются следующие форматы:
  - Строка: `"Hello, World!"` — каждый символ становится отдельным токеном
  - Список чисел: `[72, 101, 108, 108, 111]` — каждый элемент становится токеном

**Чтение из буфера:**
- Инструкция `IN Rd` читает следующий токен из буфера
- Токен извлекается из начала буфера (FIFO — первым пришёл, первым ушёл)
- После чтения токен удаляется из буфера
- При пустом буфере возвращается 0 (сигнал EOF — End Of File)
- Модель продолжает работу, но чтение возвращает 0

### Буфер вывода (Output Buffer)

**Запись в буфер:**
- Инструкция `STORE Rs1, OUT` записывает значение из регистра в буфер вывода
- Значение добавляется в конец буфера
- Буфер может хранить любое количество токенов

**Чтение из буфера:**
- По окончании моделирования все токены из буфера выводятся
- Поддерживается вывод в двух форматах:
  - Числовой: отображение числовых значений токенов
 
## Транслятор

Транслятор (`translator.py`) выполняет преобразование исходного кода на ассемблере в бинарный машинный код, готовый для загрузки в модель процессора. Трансляция выполняется за два прохода.

### Общая архитектура транслятора

Транслятор состоит из следующих основных компонентов:

| Компонент | Назначение |
|-----------|------------|
| **Лексер** | Разбивает исходный текст на токены (метки, инструкции, операнды, директивы) |
| **Парсер** | Анализирует структуру программы, выделяет секции и инструкции |
| **Таблица символов** | Хранит соответствие имён меток их адресам |
| **Генератор кода** | Преобразует инструкции в машинный код  |
| **Сборщик** | Формирует итоговый бинарный файл из секций кода и данных |

### Препроцессинг 
Перед выполнением основных проходов, транслятор делает предварительную обработку. На этом этапе обрабатываются константы, развертываются описанные макросы и блоки условной компиляции.
#### Константы
```asm
.const SIZE 10
.const DEBUG 1
```
пример объявления констант. После объявления имя константы можно использовать заместо числового литерала:
`LOADI R1 SIZE`
После обработки препроцессором, такая запись преобразуется в `LOADI R1 10`

#### Макросы
С помощью специальной директивы `.macro` можно задать последовательность инструкций, которая будет выполняться по вызову имени макроса. 
```asm
.macro ZERO reg
    LOADI reg 0
.endm
```
Например такой макрос зануляет предоставленный регистр. Для использования нужно написать `ZERO R2`. 
По такой же логике можно реализовать макросы с несколькими аргументами. Важно: вызов макросов друг из друга не поддерживается

#### Условная компиляция
Для включения или исключения определенных кусков программы, можно воспользоваться условной компиляцией, реализованной через директивы `.if`, `.ifdef`.
Например:
```asm
.const DEBUG 1

.if DEBUG
    LOADI  R1  123
.else
    LOADI  R1  999
.endif
```
В таком случае, если константа `DEBUG=1` (тождественно истина) то выполнится блок `.if`, в противном случае препроцессор включит блок `.else`.
Отдельно можно использовать директиву `.ifdef` для проверки существования константы. Тогда неважно какое значение она принимает, учитывается только факт ее объявления:
```asm
.const DEBUG 123

.ifdef DEBUG
   LOADI  R1  1
.endif
```


### Первый проход (Сбор меток и анализ структуры)

На первом проходе транслятор анализирует структуру программы без генерации машинного кода.

**Действия:**

1. **Чтение исходного файла построчно**
   - Каждая строка обрабатывается последовательно
   - Учитывается текущая позиция в памяти (адрес)

2. **Удаление комментариев**
   - Строки, начинающиеся с `;`, игнорируются полностью
   - Комментарии в конце строки (после `;`) удаляются

3. **Обработка директив**
   - `.org` <адрес>: установка текущего адреса
   - `.text`: переключение в секцию кода
   - `.data`: переключение в секцию данных
   - `.str` "<строка>": размещение Pascal-строки в памяти данных

4. **Сбор меток**
   - Метка определяется как идентификатор, заканчивающийся на `:`
   - Метка связывается с текущим адресом
   - Сохраняется в таблице символов: `{имя_метки: адрес}`

5. **Подсчёт размера секций**
   - Для секции кода: каждая инструкция занимает 1 слово (4 байта)
   - Для секции данных: `.word` занимает 1 слово, `.str` занимает (длина + 1) слов
  - Строковый: преобразование чисел в символы по ASCII

### Второй проход (Генерация кода)

На втором проходе транслятор формирует из полученной структуры бинарный файл

**Действия:**
1. **Повторное чтение исходного файла**
2. **Генерация машинного кода для каждой инструкции**:
    - Определение формата команды
    - Кодирование ее OPCODE, и операндов
    - Разрешение меток (подстановка их из таблиц)
3.** Формирование секций**:
   - Секция кода (`.text`): хранит все сгенерированные инструкции
   - Секция данных (`.data`): хранит все размещенные данные

### Формат бинарного файла
```text
┌─────────────────────────────────────────────────────────┐
│ entry_point (4 байта)                                   │
├─────────────────────────────────────────────────────────┤
│ количество секций кода (4 байта)                        │
├─────────────────────────────────────────────────────────┤
│ для каждой секции кода:                                 │
│   ├── адрес начала секции (4 байта)                     │
│   ├── количество слов в секции (4 байта)                │
│   └── машинные коды (4 байта * количество слов)         │
├─────────────────────────────────────────────────────────┤
│ количество секций данных (4 байта)                      │
├─────────────────────────────────────────────────────────┤
│ для каждой секции данных:                               │
│   ├── адрес начала секции (4 байта)                     │
│   ├── количество слов в секции (4 байта)                │
│   └── данные (4 байта * количество слов)                │
└─────────────────────────────────────────────────────────┘
```


## Модель процессора

Модель процессора (`cpu.py`) реализует поведение аппаратного обеспечения с точностью до такта (`tick`). Она симулирует выполнение машинного кода, загруженного из бинарного файла, и предоставляет механизмы для отладки и анализа работы процессора.

### Общая архитектура модели

Модель процессора состоит из следующих основных компонентов:

| Компонент | Назначение |
|-----------|------------|
| **DataPath** | Реализует путь данных: регистровый файл, АЛУ, мультиплексоры, память данных |
| **Control Unit** | Реализует микропрограммное управление: Microcode ROM, указатель микрокода, декодер |
| **Память команд (IMEM)** | Хранит исполняемый код (загружается из бинарного файла) |
| **Память данных (DMEM)** | Хранит данные и стек (инициализируется из конфигурации) |
| **Регистры** | PC, IR, SP, флаги, регистровый файл |

### DataPath (Путь данных)

DataPath объединяет все компоненты, участвующие в передаче и обработке данных.

## Руководство пользователя (Интерфейс командной строки)

Инструментальная цепочка состоит из двух основных компонентов: транслятора и модели процессора. Оба компонента управляются через командную строку.

### Транслятор

Транслятор преобразует исходный код на ассемблере в бинарный машинный код и конфигурационный файл для модели процессора.

#### Синтаксис

```bash
python translator.py <source.asm> [output_prefix]
```
В данном случае, если выходное название файла не указано, будет использовано по умолчанию `output.bin`


Пример:
```bash
python translator.py  code.asm code_binary.bin
```


### Процессор

Модель процессора выполняет машинный код из бинарного файла, используя конфигурацию для инициализации памяти и ввода-выводаб

#### Синтаксис

```bash
python main.py  <program.bin> <config.json> [options]
```
В качестве доп.аргументов могут быть переданы дополнительные настройки:
| Опция | Описание |
|-------|----------|
|--debug| Пошаговый режим работы, с выводом состояния регистров: R0-R16, флагов состояния, SP, IR, MP |
| --t   | Режим логирования, позволяет записать в файл `trace.log` полный лог исполнения программы для дальнейшей отладки|
| --s   | Вывод состояния последних пяти добавленных в стэк элементов|
| --m   | Вывод дампа памяти данных в диапазоне, указаннов в конфигуарационном файле |

### Конфигурационный файл
Для корректной работы нужно написать конфигурационный файл (`config.json`). Структурно он выглядит так:
```json
{
  "data_memory_size": ...,
  "instr_memory_size": ...,
  "entry_point": ...,
  "input": [],
  "data_image": {
    "0": 0,
    "254": 0
  },
  "instr_image": {},
  "memory": {
    "start": ...,
    "end": ...
  }
}
```
В данном случае, из конфига `data_memory_size`, и `instr_memory_size` задают модели процессора размер памяти для каждого блока отдельно. 
`entry_point` задает адрес, с которого по умолчанию начинается программа. (по умолчанию стоит 0). `input` указывает какие входные данные предоставляются программе.
`data_image`: задает начальное содержимое памяти до запуска программмы в формате: `<address>: <value>`. Аналогично с `instr_memory`. Также,  для вывода дампа памяти необходимо указать `memory`. В котором будет указан номер с какой ячейки по какую взять дамп памяти данных.

## Тестирование и CI

Для проверки корректности работы процессора и транслятора используются автоматические тесты на базе фреймворка `pytest`.

Тесты располагаются в каталоге `tests/` и запускают полный цикл работы системы:

1. Создание экземпляра процессора.
2. Загрузка конфигурационного файла (`*.json`).
3. Загрузка скомпилированной бинарной программы (`*.bin`).
4. Выполнение программы до инструкции `HALT`.
5. Проверка содержимого выходного буфера на соответствие ожидаемому результату.

Каждый тест использует отдельную программу и собственный набор входных данных. Конфигурационный файл позволяет задавать:

- размер памяти команд;
- размер памяти данных;
- начальное значение счётчика команд;
- входной поток данных;
- начальное содержимое памяти данных;
- начальное содержимое памяти команд.

Пример запуска всех тестов:

```bash
pytest tests/ -v
```
### Алгоритмы
1. **hello_world**: простой вывод строки. Используется директива .str для демонстрации поддержки строк
2. **cat**: посимвольное чтение данных из порта ввода и вывод их на экран
3. **hello_user_name**: Выводит вопрос, считывает ответ из порта ввода, и выводит ответ на вопрос в одной строке
4. **sort**: Сортировка массива простейшим алгоритмом "bubleSort". Демонстрирует возможности вызова функций, ветвлений, мат.операций.
5. **long**: Программная эмуляция арифметики двойной точности (64 бита). Реализовано через ручное использование бита переноса `carry`
6. **prob1**: Математический алгоритм поиска самого большого палиндрома, который можно получить из произведения двух трехзначных чисел (999 * 999)
